Visão geral do case

Contexto

A CaseCellShop é uma varejista fictícia especializada em capinhas para celular. O hiper crescimento elevou o volume de acessos diários de milhares para milhões e revelou gargalos na arquitetura dependente de um ERP monolítico.

Principais dores: (1) vitrine lenta por depender do ERP síncrono; (2) vendas duplicadas quando o estoque acaba; (3) checkout instável devido a timeouts do ERP ao gerar faturamento.

Objetivo do mini-projeto

Construir um fluxo de checkout resiliente que desacople as jornadas críticas do ERP, reduza latência, impeça overselling e mantenha a experiência do cliente mesmo sob falhas e picos de tráfego.

Estratégia adotada

  • APIs próprias no backend (NestJS) para vitrine e checkout.
  • Sincronização assíncrona com o ERP via BullMQ e simulador controlado de falhas.
  • Reservas transacionais de estoque + locks Redis para evitar overselling.
  • Idempotência simples baseada em Idempotency-Key.
  • Frontend em Next.js 15 com React Query, feedbacks claros e bloqueio de múltiplos cliques.
  • Observabilidade com logs estruturados, métricas Prometheus e spans OpenTelemetry.

Resultados

  • Vitrine responde em milissegundos com cache e paginação independentes.
  • Checkout confirma pedidos mesmo sob instabilidade externa e reaproveita tentativas duplicadas.
  • UX mantém mensagens claras para sucesso, validação, estoque insuficiente e falha temporária.
  • Documentação, testes automatizados e automações Docker prontos para evolução incremental.

Arquitetura e design

Visão macro

Next.js (FSD) → NestJS (DDD/Clean) → PostgreSQL / Redis → BullMQ + simulador ERP.

Camadas principais

  • Frontend: cenas, widgets e features com FSD; React Query, Redux persistido, Tailwind e Framer Motion.
  • Backend: módulos DDD; validação com DTOs; Prisma para Postgres; Redis para cache, locks e idempotência.
  • Fila e simulador: BullMQ sincroniza o ERP de forma assíncrona, simulando latência/falhas com circuito de proteção.

Fluxos críticos

  1. Vitrine: `GET /products` paginado, com busca e cache Redis.
  2. Checkout: `POST /checkout` valida payload, aplica lock Redis, registra pedido e retorna 202 com idempotência.
  3. Sincronização ERP: worker atualiza o status (PENDING → SUCCESS/FAILED) com retries controlados.

Back-end

Construído em NestJS 10 com Prisma. Endpoints principais: `GET /products`, `POST /checkout` (idempotente) e `GET /orders/:id`. Locks Redis evitam concorrência de estoque. Respostas distintas para validação (`400`), conflito de estoque (`409`), falta de idempotência (`422`) e falhas técnicas (`500`).

A sincronização com o ERP acontece em segundo plano via BullMQ; o simulador pode gerar sucesso imediato, falha temporária (retry com backoff) ou falha definitiva.

Front-end

Aplicação Next.js 15 preparada para modo App Router. A vitrine usa React Query, skeletons e fallback de imagem. O checkout bloqueia multi-cliques, exibe toasts claros e mantém o estado coerente após erro/retry. O painel `/admin` oferece filtros por status e badges traduzidas.

Qualidade e operações

  • Scripts de execução local (`npm run dev` no frontend, `npm run start:dev` no backend).
  • Workflow com Docker Compose provisionando Postgres, Redis, API e UI.
  • Suite de testes (unit, integration, e2e) documentada no README.
  • Observabilidade: logs Pino, métricas em `/metrics` e spans OpenTelemetry.

Parte 1.A — Perguntas conceituais

  1. Desacoplamento do ERP via camada intermediária (Checkout Service).
  2. Consistência de estoque com transações e locks Redis.
  3. Resiliência usando filas BullMQ, retries e circuito.
  4. Idempotência por header `Idempotency-Key` persistido em Redis.
  5. Organização DDD/Clean para evolução segura.
  6. Próximos passos: CDC, observabilidade distribuída, compensações e dashboards.
