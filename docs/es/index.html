Visión general del case

Contexto

CaseCellShop es una minorista ficticia de fundas para celulares. El hipercrecimiento incrementó el tráfico diario de miles a millones y puso en evidencia los límites de una arquitectura dependiente de un ERP monolítico para vitrina y checkout.

Dolencias principales: (1) vitrina lenta debido a llamadas síncronas al ERP; (2) ventas duplicadas cuando el stock llega a cero; (3) checkout frágil por timeouts del ERP al generar pedidos.

Objetivo

Construir un flujo de checkout resiliente que desacople las jornadas críticas del ERP, reduzca latencia, evite overselling y mantenga la confianza del cliente incluso ante fallas externas.

Estrategia

  • APIs dedicadas en NestJS para catálogo y checkout.
  • Sincronización asíncrona con el ERP usando BullMQ y un simulador de latencia/fallas.
  • Reservas transaccionales de inventario + locks distribuidos en Redis.
  • Idempotencia simple mediante el header Idempotency-Key persistido en Redis.
  • Frontend en Next.js 15 con React Query, mensajes claros y bloqueo de múltiples clics.
  • Observabilidad con logs estructurados, métricas Prometheus y spans OpenTelemetry.

Resultados

  • Vitrina responde en milisegundos con cache y paginación independientes del ERP.
  • Checkout confirma pedidos bajo inestabilidad y reutiliza intentos duplicados con seguridad.
  • Mensajes claros para éxito, validación, stock insuficiente y fallos temporales.
  • Documentación, pruebas y automatización preparadas para evolución continua.

Arquitectura y diseño

Vista macro

Next.js (FSD) → NestJS (DDD/Clean) → PostgreSQL / Redis → BullMQ + simulador ERP.

Capas principales

  • Frontend: componentes orientados a escenas, cache con React Query, carrito persistido, Tailwind CSS y Framer Motion.
  • Backend: módulos NestJS con DTOs, Prisma ORM, Redis para cache/locks/idempotencia, workers BullMQ.
  • Cola y simulador: jobs asíncronos que emulan latencia/fallas con protección mediante circuit breaker.

Flujos críticos

  1. Catálogo: `GET /products` paginado, con búsqueda y cache Redis.
  2. Checkout: `POST /checkout` valida payloads, aplica lock Redis, registra el pedido y responde 202 con idempotencia.
  3. ERP: worker convierte pedidos de PENDING a SUCCESS/FAILED con reintentos controlados.

Back-end

Construido con NestJS 10 y Prisma. Endpoints clave: `GET /products`, `POST /checkout` (idempotente) y `GET /orders/:id`. Redis protege los decrementos de inventario. El sistema diferencia errores de validación (400), conflicto de stock (409), falta de idempotencia (422) y fallos técnicos (500).

La sincronización con el ERP ocurre en background a través de BullMQ; el simulador puede devolver éxito inmediato, fallo temporal con backoff o fallo definitivo.

Front-end

Aplicación Next.js 15 que utiliza React Query, skeletons y fallback de imágenes. El checkout evita múltiples envíos, los toasts comunican cada estado y el panel `/admin` ofrece filtros y badges localizados.

Calidad y operaciones

  • Scripts locales (`npm run dev`, `npm run start:dev`) y Docker Compose para todo el stack.
  • Matriz de pruebas (unitarias, integración, e2e) descrita en el README.
  • Observabilidad con logs Pino, endpoint `/metrics` y trazas OpenTelemetry.

Parte 1.A — Preguntas conceptuales

  1. Desacoplar el ERP mediante un checkout service intermedio.
  2. Garantizar consistencia con transacciones SQL y locks Redis.
  3. Manejar la lentitud del ERP con BullMQ, reintentos y circuito de protección.
  4. Idempotencia con el header `Idempotency-Key` almacenado en Redis.
  5. Organizar el código con DDD/Clean para evolucionar con seguridad.
  6. Próximos pasos: CDC del ERP, observabilidad distribuida, compensaciones y dashboards.
