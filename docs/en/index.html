Project overview

Context

CaseCellShop is a fictional retailer of phone cases. Hyper growth pushed daily traffic from thousands to millions while the storefront relied on a monolithic ERP for catalog and checkout, exposing severe bottlenecks.

Key pain points: (1) slow storefront due to synchronous ERP calls; (2) overselling when multiple customers buy the last units; (3) unreliable checkout because ERP timeouts cancel purchases.

Goal

Deliver a resilient checkout flow that decouples critical journeys from the ERP, reduces latency, prevents overselling, and preserves user trust even under peak load or external failures.

Strategy

  • Dedicated NestJS APIs for catalog and checkout.
  • Asynchronous ERP synchronization using BullMQ and a configurable failure/latency simulator.
  • Transactional stock reservations plus Redis distributed locks to eliminate overselling.
  • Simple idempotency via the Idempotency-Key header stored in Redis.
  • Next.js 15 frontend with React Query, clear feedback for every state, and double-click protection.
  • Observability with structured logs, Prometheus metrics, and OpenTelemetry spans.

Outcome

  • Storefront responses in milliseconds thanks to caching and pagination detached from the ERP.
  • Checkout acknowledges orders despite ERP instability and safely replays duplicate attempts.
  • UX delivers precise messaging for success, validation issues, low stock, and transient failures.
  • Documentation, automated tests, and Docker automation ready for incremental evolution.

Architecture & design

High-level picture

Next.js (FSD) → NestJS (DDD/Clean) → PostgreSQL / Redis → BullMQ + ERP simulator.

Core layers

  • Frontend: scene-driven components, React Query caching, persisted Redux cart, Tailwind CSS, and Framer Motion microinteractions.
  • Backend: modular NestJS with DTO validation, Prisma ORM, Redis for cache/locks/idempotency, BullMQ workers.
  • Queue & ERP simulator: asynchronous jobs emulate realistic latency/failure patterns guarded by a circuit breaker.

Critical flows

  1. Catalog: `GET /products` paginated with search and Redis caching.
  2. Checkout: `POST /checkout` validates payloads, applies a Redis lock, commits the order, and returns 202 with idempotent behaviour.
  3. ERP sync: worker transitions orders from PENDING to SUCCESS/FAILED with controlled retries.

Back-end

Built with NestJS 10 and Prisma. Main endpoints: `GET /products`, `POST /checkout` (idempotent), and `GET /orders/:id`. Redis locks protect stock decrements. Response codes distinguish validation errors (400), stock conflicts (409), missing idempotency (422), and technical failures (500).

ERP synchronization happens off the critical path through BullMQ; the simulator can return immediate success, transient failure (with exponential backoff), or definitive failure.

Front-end

Next.js 15 application leveraging React Query, suspense-friendly loading states, and image fallbacks. Checkout buttons guard against accidental multiple submissions, toasts explain every outcome, and the `/admin` dashboard supplies filters and localized status badges.

Quality & operations

  • Local scripts (`npm run dev`, `npm run start:dev`) plus Docker Compose for full environment.
  • Test matrix (unit, integration, e2e) documented in the README.
  • Observability stack: structured Pino logs, `/metrics` for Prometheus, OpenTelemetry traces.

Part 1.A — Conceptual answers

  1. Decouple the ERP via an intermediate checkout service.
  2. Guarantee stock consistency with SQL transactions plus Redis locks.
  3. Handle ERP slowness through BullMQ retries, circuit breaker, and background workers.
  4. Implement idempotency via the `Idempotency-Key` header stored in Redis.
  5. Organise the project with DDD/Clean architecture for safer evolution.
  6. Next steps: ERP CDC, distributed observability, compensation flows, rate limiting, dashboards.
